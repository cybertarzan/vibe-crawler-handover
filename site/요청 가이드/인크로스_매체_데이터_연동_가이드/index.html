    인크로스 매체 데이터 전송 및 프로젝트 연동 가이드

    본 문서는 Vibe Crawling Platform에서 인크로스(Incross) 매체 데이터를 시스템 내부적으로 어떻게 관리하고, 1차 DataLake API로 전송할 때 어떠한 매핑 규칙을 사용하는지 설명하는 가이드입니다.

    1. 개요

    인크로스 API 연동을 통해 수집된 데이터 중 매체(Media) 정보는 시스템 내부적으로 '카테고리(Category)' 테이블에서 통합 관리됩니다. 프로젝트 생성 시 선택하는 카테고리 값이 실제로는 해당 캠페인의 집행 매체 정보를 담고 있습니다.

    백엔드 시스템은 프로젝트에 설정된 매체 정보가 유실되지 않도록, 최종 데이터 전송 단계에서 카테고리 정보를 매체 정보로 자동 치환하여 전달합니다.

    2. 데이터 매핑 로직

    2.1. 데이터 소스 (Source of Truth)

    시스템 내에서 매체 정보의 신뢰할 수 있는 단일 원천(Source of Truth)은 카테고리(Category) 데이터입니다.

    구분 시스템 필드 (Internal) 외부 전송 필드 (DataLake API) 비고
    매체 ID category_code mediaid 실제 인크로스 매체 고유 코드
    매체 명 category_name medianame 매체 서비스 명칭 (예: Google, DA 등)

    2.2. 전송 단계 바인딩 규칙

    크롤링 작업 완료 후 1차 DataLake API로 메타데이터를 전송할 때 다음과 같은 우선순위로 매체 정보를 조립합니다.

    1. 카테고리 정보 우선 참조: 프로젝트에 연결된 카테고리의 category_code가 존재할 경우, 이를 mediaid로 사용합니다.
    2. 프로젝트 필드 보완: UI상의 제약(캠페인 수동 입력 등)으로 프로젝트 테이블(projects)의 미디어 필드가 비어있더라도, 카테고리가 선택되어 있다면 백엔드 아키텍처 수준(Repository Join)에서 이를 찾아내어 자동으로 바인딩합니다.

    3. 개발 및 운영 가이드

    3.1. 프로젝트 생성 시 주의사항

    현재 UI 구조상 캠페인 정보를 수동으로 입력하더라도, 카테고리(매체) 항목은 반드시 올바르게 선택해야 합니다. - 카테고리 선택 시 백엔드에서 해당 카테고리에 매핑된 실제 인크로스 매체 식별자를 찾아 전송하기 때문입니다. - 카테고리를 잘못 선택할 경우, 1차 API 전송 시 엉뚱한 mediaid가 전달되어 집계 데이터 오염의 원인이 됩니다.

    3.2. 백엔드 구현 참조

    매체 정보 바인딩 로직은 다음 구성 요소를 통해 구현되어 있습니다. - Repository: ProjectRepositoryAdapter에서 조회 시 CategoryModel을 항상 Outer Join하여 매체 코드를 로드. - Service: start_crawling_service.py 내 project_meta_binding 로직에서 카테고리 코드를 mediaid로 직결 매핑.

    4. 장애 대응 및 데이터 정정

    만약 특정 프로젝트의 매체 정보가 NULL로 전송되거나 잘못된 매체로 집계되고 있다면 다음 사항을 점검하십시오. 1. 해당 프로젝트의 category_id가 올바른 매체 카테고리를 가리키고 있는지 확인. 2. categories 테이블의 category_code 필드값이 실제 인크로스 매체 코드와 일치하는지 확인.
