# 스토리지 용량 계획

> 기준일: 2026-03-17 (운영서버 96GB 디스크)

---

## 1. 세션 로그 구성 요소별 용량 비율
크롤링 스케줄이 구동될 때 생성되는 개별 세션 로그는 다음의 비중으로 구성됩니다.

| 구성 요소 | 용량 단위 | 목적 및 설명 |
|---|---|---|
| `action_snapshots/` | GB 단위 | 단계별 과정 추적용 고화질 PNG 스크린샷 (가장 큰 용량 차지) |
| `videos/` | GB 단위 | 화면 동작 흐름 전체 녹화본 (webm) |
| `downloads/` | MB 단위 | 실제 타겟 애플리케이션으로 넘길 수집 결과 (CSV 등) |
| `crawling_result.json` | KB 단위 | 결과 확인을 위한 메타데이터 |

---

## 2. 자동화된 스토리지 최적화 (Cleanup Batch Job)

현재 크롤링 플랫폼은 저장 공간의 한계가 오지 않도록 **지속적인 백그라운드 자동화 배치 작업**을 통해 디스크 공간을 관리하고 있습니다. 

*   **10일 자동 파기 설정**: 시스템 내부 스케줄러(`cleanup_old_logs_runner`)가 매일 새벽에 구동되어, **10일이 경과한 오래된 세션 데이터(스크린샷, 비디오, 덤프)를 자동으로 파기**합니다.
*   **넉넉한 디스크 여유량 보장**: 위와 같은 구조로 인해 끊임없이 새로운 파일 덤프가 쌓이더라도, 가장 큰 용량을 차지하는 시각적 디버깅 데이터가 즉각 만료 처리되므로, 현재 **개발/운영서버 환경에서는 디스크 풀(Full) 장애 위험 없이 스토리지 용량이 생각보다 매우 넉넉하게 유지**되고 있습니다.

> **Note**: 별도의 서버 증설이나 값비싼 SSD 스토리지 확장이 없다 하더라도 현재의 삭제 배치 룰만 유지하신다면 50~100GB 수준의 기본 스토리지 환경에서도 원활한 운영이 가능합니다.

## 용량 모니터링

```bash
# 운영서버 디스크 확인
df -h /

# 워크스페이스별 용량
du -sh ~/vibe-deployment/system_storage/workspaces/*/

# 날짜별 세션 용량 (전체 프로젝트 합산)
for d in ~/vibe-deployment/system_storage/workspaces/*/projects/*/sessions/*/; do
  echo "$(basename $d) $(du -sm $d 2>/dev/null | cut -f1)MB"
done | awk '{a[$1]+=$2} END{for(d in a) print d, a[d]"MB"}' | sort
```

> **주의**: 디스크 사용률 80% 초과 시 디스크 증량 또는 보관 기간 축소를 검토해야 합니다.
> 디스크 100% 도달 시 크롤링 스크린샷 저장 실패 → 무한루프 발생 위험이 있습니다.
