# 최종 개발 견적서 (ISS-003 백그라운드 스케줄러 & 워커 노드 분산)

이 견적서는 Vibe AI Crawler 안정화를 위해 FastStream 기반 마스터-워커 분산 아키텍처를 도입하는 데 필요한 개발 범위, 공수, 그리고 최종 견적을 통합한 문서입니다.

---

## 1. 요구사항 및 기능 명세

외부 Worker 노드 추가 및 FastStream/Redis 비동기 큐 아키텍처 도입 시, 기존 단일 모놀리식 서버 구조에서 동작하던 백그라운드 스케줄러들을 통제하고 분리하는 상세 내역입니다.

### 1.1 마스터 중심 스케줄러 (Master Only)
오로지 180 서버(API 서버)에서만 작동하며 스케줄 생성과 관제를 담당.
- **체크 및 실행 서비스**: 크론(Cron) 주기를 판단하여 PENDING 상태로 만들고, 도커를 직접 실행하지 않으며 발생 즉시 **Redis 큐로 Push**.
- **관제 서비스 (복구/동기화)**: 워커가 가져간 작업이 장시간 RUNNING 상태로 멈춰 있을 때, 타임아웃을 감지하여 `FAILED` 처리하거나 `DLQ(Dead Letter Queue)`로 보내는 기반 설계.

### 1.2 워커 중심 스케줄러 (Worker Only)
신규 투입될 외부 서버에서 자체적으로 동작하는 스케줄러.
- **FastStream Consumer**: Redis를 감청하다가 워커가 허용하는 슬롯(예: 5개)만큼 큐에서 가져와 순차적으로 컨테이너 실행.
- **클린업 크론**: 비정상적으로 남은 컨테이너 고아(Orphan) 데이터를 삭제하고, 전송 완료된 로컬 스토리지 파일을 청소하여 스스로 용량을 유지.

### 1.3 인프라 및 통신 강건화(Robustness) 설계
- **보안/스토리지**: 외부 워커망 통신을 위한 방화벽 세팅 및 마스터 내 Alpine Redis 큐잉 서버 탑재. 워커는 독립적인 로컬 SSD 스토리지 사용.
- **버퍼/보상망**: 워커-마스터 데이터 통신이 끊어졌을 때 PENDING 작업들을 DB Buffer에 계속 쌓고 차후 복구하는 통신 데드서킷 차단망 설계.

## 4. 수행 일정 계획 (총 14.5 영업일; 약 3주 기간)

1. **기반공사 및 세팅 (1일차)**: 외부 워커망 방화벽/Redis 세팅 및 Docker Compose 분할 구축.
2. **Master 코어 파트 (2 ~ 7일차)**: 관제 스케줄러 레거시 분리, FastStream Publisher 개편, Edge 복구망(API) 설계 진행.
3. **Worker 에이전트 파트 (8 ~ 9일차)**: 워커측 Consumer 로직(감청 및 컨테이너 제어/클린업) 구축.
4. **집중 QA 및 시스템 테스트 (10 ~ 14일차)**: 단위/교차 통합 테스트 집중 검증 및 워커 다중 세션 적체/버퍼 시나리오의 런타임 안정성 검수.
